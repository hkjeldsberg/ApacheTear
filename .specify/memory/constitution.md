# Apache Tear Constitution

## Core Principles

### I. Code Quality

TypeScript strict mode is MANDATORY across the entire codebase — `any` is
forbidden and all functions MUST declare explicit return types. Interactive
UI components MUST have component-level tests. Files MUST NOT exceed 300 lines;
split into smaller modules when the limit is reached. New features MUST NOT
modify core/shared modules unless strictly necessary. All commits MUST follow
[Conventional Commits](https://www.conventionalcommits.org/) format
(`feat:`, `fix:`, `chore:`, `docs:`, `style:`, `refactor:`, `test:`), and
a commit MUST be made at the end of each phase and after each major feature.

**Rationale**: Strict typing and module limits prevent technical debt from
accumulating silently. Conventional commits enable automated changelogs and
reliable CI/CD gates.

### II. Performance

No blocking third-party scripts are permitted in the critical render path.
Server components MUST be the default choice; client components are permitted
only where interactivity strictly requires them.

**Rationale**: Next.js App Router's server-first model delivers faster
Time-to-First-Byte and reduces client-side JavaScript bundle size, directly
improving perceived load time and Core Web Vitals scores.

### III. Accessibility & Universal Design

All UI MUST meet or exceed the following requirements:

- Color contrast: ≥ 4.5:1 for normal text, ≥ 3:1 for large text (WCAG AA)
- Images MUST carry descriptive `alt` text; decorative images MUST use `alt=""`
- Every form field MUST have an associated `<label>` element —
  placeholder-only labels are forbidden
- Error messages MUST be descriptive and suggest a corrective action
- Touch targets MUST be at minimum 44×44 px on mobile viewports
- No content MAY flash more than 3 times per second
- Responsive layout MUST support viewports from 375 px (mobile) to 1440 px+
  (desktop)

**Rationale**: Accessibility is non-negotiable for a personal knowledge tool
intended for use across devices and by users with varying abilities.

### IV. Security & Privacy (GDPR)

Authentication MUST be handled exclusively via Supabase Auth — no custom
session management is permitted. Session tokens MUST NEVER be exposed
client-side or committed to version control. Data collection MUST be minimal
and purposeful, limited strictly to what the application functionally requires.
`.env.local` MUST be gitignored; all environment variables MUST be documented
in `.env.example`.

**Rationale**: Personal knowledge data is sensitive. Delegating auth to
Supabase eliminates a class of session-handling vulnerabilities, and minimal
data collection limits GDPR exposure and attack surface.

### V. Maintainability

The folder structure MUST follow Next.js App Router conventions. Shared UI
components MUST reside in `components/`, business logic in `lib/`, and database
queries in `lib/db/`. Page and route files MUST contain no business logic —
all logic MUST be delegated to `lib/` functions. All Supabase queries MUST be
typed using types generated by `supabase gen types`.

**Rationale**: Consistent, predictable structure reduces onboarding friction
and prevents logic from spreading across layers in ways that complicate testing
and refactoring.

## Governance

This constitution supersedes all other practices and ad-hoc conventions within
the project. Any amendment MUST be reflected in an updated version of this file
and communicated via a `docs:` commit.

**Amendment procedure**:

1. Propose the change with a rationale referencing an observed gap or new
   requirement.
2. Update `.specify/memory/constitution.md` with the revised content.
3. Increment `CONSTITUTION_VERSION` per semantic versioning rules:
   - MAJOR: principle removal, redefinition, or backward-incompatible
     governance change.
   - MINOR: new principle or section added, or materially expanded guidance.
   - PATCH: clarification, wording fix, or non-semantic refinement.
4. Run the consistency propagation checklist against all dependent templates.
5. Commit with: `docs: amend constitution to vX.Y.Z (<summary>)`

**Compliance review**: All PRs MUST be verified against the constitution
before merge. The "Constitution Check" gate in `plan-template.md` MUST be
passed before Phase 0 research begins on any feature.
